# Diary Response

读 Obsidian 日记里的 `## 碎念流`，然后在 `## 长文` 下写一篇 GPT 对这些碎碎念的整体回应。

这个 skill 的重点不是整理、总结、复述，而是回应。

## Target note

- 如果用户给了当前笔记路径，用该路径。
- 如果用户给了日期，用 `01日记/YYYY-MM-DD.md`。
- 如果是自动化运行且没有指定路径/日期，处理昨天的日记。
- 00:00 运行时，处理昨天，而不是当天。
- macOS 计算昨天：

```bash
date -v-1d +%F
```

然后目标路径是：`01日记/<date>.md`。

## Workflow

1. 读取目标笔记。
2. 找到 `## 碎念流` 和 `## 长文`。
3. 使用 `## 碎念流` 作为当天回应的来源。
4. 如果 `## 碎念流` 里有本地图片嵌入，例如 `![[image.jpg]]`，并且当前工具能读图，就把图片也当作碎念材料；读不了不要编。
5. 可以在 `## 碎念流` 原文上添加少量 gpt Hover Reveal 批注，但不是必须。
6. 在 `## 长文` 下写 GPT 的整体回应。
7. 替换原有 `## 长文` 内容。
8. 不要删除或重写 `## 碎念流` 原文；如果添加批注，只能轻微添加。

如果文件不存在、`## 碎念流` 为空、或缺少 `## 长文`，停止并说明原因。

## Optional annotations on 碎念流

如果添加批注，使用 Hover Reveal 语法：

```md
[原文中可见的句子或短语]{gpt：批注内容}
```

中括号里的内容本身就是正文可见内容，不要重复写两遍。

批注要求：

- 少量即可。
- 不需要每条都批。
- 可以是喜欢、不同意、联想、提醒、吐槽、轻轻补一句。
- 简单自然，不要用力证明自己聪明，不要油，不要像心理咨询或阅读理解。
- 禁止重复或复述原文。
- 如果不知道说什么，就不要批注。

好的批注例子：

```md
[今天的风有点冷，但我居然很想把它记下来。]{gpt：这个句子很轻，但它留下的是那一秒真的存在过。}
```

## Writing 长文

`## 长文` 是 GPT 在读完 `## 碎念流` 后写给用户的整体回应。

它不是：

- 不是用户第一人称日记；
- 不是对碎念流的整理；
- 不是逐条总结；
- 不是素材分类；
- 不是心理咨询；
- 不是正式书信。

它可以：

- 回应 GPT 真正读到的东西；
- 说出文本下面隐约在动的情绪、问题、欲望、矛盾；
- 帮用户看见一些可能还没完全看见的线索；
- 提出下一天或下一周可以考虑、尝试、创造、暂时放过的东西；
- 有联想、有一点判断、有一点温度。

硬规则：

- 不要按 `## 碎念流` 的顺序一段段写。
- 不要写“关于 A / 关于 B / 关于 C”。
- 不要把一天包装成一个漂亮主题。
- 不要反复引用或点名某条碎念。
- 不要把长文写成加长版批注。
- 不要为了显得懂而用力证明自己聪明。
- 如果写不出很有灵气的东西，宁可朴素。

成功标准：

- 读起来像 GPT 完整读过这一天，然后写了一篇回应。
- 不像 GPT 在按素材顺序点评。
- 不像 GPT 在总结用户今天说了什么。

## Completion

完成后简单回复：编辑了哪个笔记，说明已处理 `## 长文`，以及是否添加了 `## 碎念流` 批注。不要把全文贴回聊天，除非用户要求。
