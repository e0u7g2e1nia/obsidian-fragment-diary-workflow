# 设计迭代记录：从“自动扩写日记”到“GPT 整体回应”

这个工作流不是一开始就长成现在这样的。它经历过几次重要调整。

## 1. 最初版本：把碎念整理成第一人称长文

最初的想法是：

- 读取当前日记的 `## 碎念流`；
- 把零散碎碎念整理、扩写成 `## 长文`；
- 正文要模仿用户本人的写作和说话风格；
- GPT 的评价不直接插入正文，而是用 Hover Reveal 批注。

问题是：模型很容易把它写成总结、报告、刻意的时间线，或者一种“AI 假装是用户”的文字。

## 2. 批注系统：保留，但要轻

Hover Reveal 批注本来是为了让 GPT 的回应不要打断正文。

后来发现批注也会出问题：

- 有些批注太像阅读理解；
- 有些批注太用力、假装懂；
- 有些批注重复原文；
- 有些被批注的句子会重复显示；
- 手机端显示会很丑。

所以最后的规则变成：少量、自然、可有可无。

## 3. 用户真正想要的不是“整理”，而是“被回应”

后来需求变成：

- GPT 直接在 `## 碎念流` 上少量批注；
- `## 长文` 不再写成用户第一人称日记；
- `## 长文` 变成 GPT 读完整天碎念之后写下的整体回应。

这个回应应该能：

- 看见用户真正关心的东西；
- 挖出文本下面还没完全显出来的问题；
- 有联想、有判断、有一点轻轻拽住；
- 给出第二天或下一周可以考虑、尝试、创造、暂时放过的东西。

## 4. 最重要的反面规则

不要：

- 逐条总结；
- “关于 A / 关于 B / 关于 C”；
- 把一天包装成一个主题；
- 写成心理咨询；
- 写成正式书信；
- 用很多漂亮但空的判断证明自己懂了。

如果写不出灵气，宁可朴素。

## 5. 自动化原则

真正的自动化 prompt 要短。

复杂写作规则放在 skill 里。自动化只负责：

- 什么时候运行；
- 处理哪一天；
- 调用哪个 skill；
- 是否写回文件。

这能减少自动化 prompt 和 skill 互相打架。
